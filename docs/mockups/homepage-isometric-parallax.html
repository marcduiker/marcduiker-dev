What is publiccode.yml?

A small, standardised metadata file that describes software built for and by public administrations — making it discoverable, reusable, and easy to share between governments.

Public administrations around the world keep solving the same problems: a permit system here, a citizen portal there, a scheduling tool somewhere else. Much of that software is built with public money, yet it stays invisible — locked inside one organisation, impossible to find or reuse by the next team facing the exact same need.

publiccode.yml is a deceptively simple answer to that problem. It's a machine-readable manifest you drop in the root of a repository to describe what the software is, who maintains it, and how it can be reused. Think of it as a package.json for public-sector software.

A minimal example

The file is plain YAML. A small but valid manifest looks like this:

# publiccode.yml — placed in the repository root
publiccodeYmlVersion: "0.4"
name: My Public Service
url: "https://github.com/example/my-public-service"
releaseDate: "2026-06-24"
platforms:
  - web
  - linux
categories:
  - it-development
developmentStatus: stable
softwareType: standalone/web
description:
  en:
    shortDescription: A reusable service for public administrations.
    features:
      - Open standards
      - Accessible by default
legal:
  license: EUPL-1.2
maintenance:
  type: community

Why public administrations use it

The format is intentionally boring — and that's the point. Because it's standardised, tools can read it automatically:

  • Discoverability. Crawlers index repositories that contain a manifest and surface them in public software catalogues, so other administrations can find what already exists.
  • Reuse. In several countries, reusing existing public software is a legal default. A consistent manifest makes "is there already something like this?" an answerable question.
  • Transparency. Licensing, maintenance status, and the responsible body are declared up front, in one place.
  • Interoperability. Categories and platform fields use a shared vocabulary, so listings stay comparable across thousands of projects.
“If it is public money, it should be public code as well.” — the Public Money, Public Code campaign

Getting started

Adoption is refreshingly low-effort: add a publiccode.yml to your repository root, fill in the required fields, and validate it against the published schema. The official standard documents every field, and editors and validators exist to catch mistakes before a crawler ever sees the file.

It won't transform how your team writes code — but it makes the software you've already built findable, and that's often the missing link between "we built this" and "someone else reused it."
